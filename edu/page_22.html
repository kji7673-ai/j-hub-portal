통합문서관리 플랫폼 기술 및 교육 가이드: RAG와 클라우드

통합문서관리 플랫폼 기술 및 교육 가이드

작성자: (주)진양엔지니어링건축사사무소 김중일 대표이사
작성일자: 2026-07-17

1. 상황 설명 및 기능 도입 배경 📖

최근 진행 중인 [A구역] 정비사업이나 [B지구] 재개발 심의와 같은 대형 프로젝트에서, 수많은 법규와 심의 의견이 파편화되어 관리되고 있었습니다. 도면을 열어 의견을 수기로 작성하고, 지난 번 다른 프로젝트에서 지적받았던 법규 사항을 다시 찾아보는 과정은 매우 소모적이었습니다.

이러한 비효율을 해결하기 위해 우리는 "통합문서관리 플랫폼"을 구축하였습니다. 이 플랫폼은 단순히 PDF를 띄워주는 것에 그치지 않고, 인공지능(AI)이 직접 도서를 읽고 숨은 위험 요소를 스캔해 줍니다. 이번 업데이트에서는 여기서 한 걸음 더 나아가 RAG(검색 증강 생성)라는 최신 AI 기술을 도입하여, 과거 [C구역] 등에서 축적한 '정비사업 통합검토 보고서'를 AI가 먼저 학습한 뒤 맞춤형 피드백을 내놓을 수 있도록 시스템을 진화시켰습니다.

2. RAG 기반 분석 기능이란? 🧠

2.1 RAG (Retrieval-Augmented Generation) 개념
AI 모델은 세상의 일반적인 지식은 똑똑하게 알지만, 우리 회사(진양엔지니어링)만의 내부 프로젝트 정보는 모릅니다. RAG 기능은 AI가 대답을 하기 전에, 우리 회사의 [과거 프로젝트 검토 보고서]를 먼저 검색(Retrieval)하여 읽게 만든 다음, 그 정보를 바탕으로 대답(Generation)하게 만드는 기술입니다.

2.2 RAG 적용에 따른 분석의 차별성
기존에는 "건축법상 인동간격은 0.5배입니다"라고 원론적인 대답만 했다면, RAG 도입 후에는 "우리가 이전에 수행한 [ID-a8f9] (구 [A구역]) 보고서에 따르면 0.8배로 강화하여 적용한 선례가 있으니 이번 심의 도서도 0.8배로 검토해야 합니다"와 같이 사내 노하우가 결합된 날카로운 지적(레드팀 피드백)을 수행합니다.

3. 백엔드(Backend)와 프론트엔드(Frontend)의 역할 분담 🏗️

플랫폼은 크게 사용자가 보는 '프론트엔드'와 보이지 않는 곳에서 일하는 '백엔드'로 나뉩니다.

용어 해설: 프론트엔드 vs 백엔드

  • 프론트엔드 (Frontend): 우리가 브라우저(크롬, 사파리)를 통해 보는 화면입니다. 버튼, 메뉴, PDF 뷰어 등 사용자의 클릭을 받아들이는 '창구' 역할을 합니다. (예: React, Vite)
  • 백엔드 (Backend): 프론트엔드에서 버튼을 눌렀을 때, 그 요청을 받아 실제로 데이터를 처리하는 '두뇌'이자 '공장'입니다. AI에게 문서를 보내고, PDF에서 텍스트를 추출하며, 서버에 파일을 저장하는 무거운 작업은 모두 백엔드가 수행합니다. (예: Python, FastAPI)

3.1 흐름도 (Architecture Flow)

graph LR User([사용자]) -- "클릭/업로드" --> FE[프론트엔드
(React UI 화면)] FE -- "API 호출 (문서/데이터)" --> BE[백엔드
(FastAPI 두뇌)] BE -- "프롬프트 + RAG 데이터" --> AI[클라우드 AI
(Gemini 3.5 Flash)] AI -- "분석 결과" --> BE BE -- "결과 전송" --> FE FE -- "화면 출력" --> User

4. 클라우드 서버 이용에 따른 차별성 ☁️

단순한 '정적 웹페이지'는 정보를 보여주기만 할 뿐 상호작용이 불가능합니다. 하지만 우리는 클라우드 서버(Cloud Server) 및 AI API를 결합하여 차원이 다른 플랫폼을 구성했습니다.

  1. 어디서든 접근 가능: 사무실의 맥북이 아니더라도, 인터넷이 연결된 아이패드나 스마트폰에서도 클라우드 주소로 접속하면 즉시 업무를 볼 수 있습니다.
  2. 무한한 연산 자원: PDF 수백 장의 텍스트를 읽고 AI가 분석하는 데는 엄청난 컴퓨터 자원이 필요하지만, 이 무거운 작업을 모두 클라우드(Google Gemini API 서버)에서 대신 처리해주므로 내 기기는 느려지지 않습니다.
  3. 지속적인 데이터 자산화: 개개인의 PC에 흩어져 있던 도면과 의견서가 클라우드 스토리지(데이터베이스)에 중앙 집중식으로 저장되므로, 어떤 담당자가 열어보아도 동일한 퀄리티의 RAG 지식으로 재활용됩니다.

5. 보안 및 가명화 (블라인드) 준수 사항 🔒

클라우드와 AI를 사용함에 있어 가장 중요한 것은 보안입니다.

보안 위반 주의
도면 파일 그대로를 외부 AI 서버에 넘기면 설계 기밀과 조합원 정보가 유출될 위험이 있습니다.

우리의 백엔드는 AI에게 데이터를 보내기 직전, 아래와 같은 보안 가명화 파이프라인을 거치도록 설계되어 있습니다:

  1. 정규식 매칭 (Regex): 주민등록번호, 연락처 등을 찾아 *** 로 마스킹 처리
  2. 도면 식별자 블라인드: 조합명이나 주요 식별자가 도면에 있을 경우, 서버 메모리 상에서 텍스트를 즉각 파기하거나 가명(예: [A구역])으로 치환한 후 전달
  3. Session 인증: 플랫폼 자체도 인가된 사내 이메일(예: ceo.jinyang@...)로 구글 로그인(OAuth)을 거친 사용자만 접근할 수 있도록 이중 잠금 되어 있습니다.

6. 사전 준비 및 환경 구성 가이드 🛠️

업무 인수인계를 하거나 새로운 PC에서 이 플랫폼을 띄워보려면 아래의 환경 구성이 필요합니다.

법규관련/
 ├── DocReviewPlatform/
 │    ├── frontend/       (UI 소스코드 및 패키지)
 │    ├── backend/        (Python API 및 AI 연동 코드)
 │    ├── run.sh          (전체 시스템 원클릭 구동 스크립트)
 │    └── .env            (🔒 Gemini API 키 등 보안 비밀번호 보관)
 ├── 의뢰프로젝트/          (RAG에 참조할 보고서 마크다운 저장소)
 └── 생성 자료/
      └── 2026-07-17/
           └── 교육자료_매뉴얼/   (지금 보고 계신 이 문서의 위치)

7. 프롬프트 실전 예시 💬

플랫폼에 RAG 기능을 더 잘 활용하려면, AI가 올바른 컨텍스트를 이해할 수 있도록 프롬프트를 주는 기술(프롬프트 엔지니어링)이 필요합니다. 플랫폼 내부 백엔드에는 이미 다음과 같은 프롬프트가 탑재되어 있습니다.

[AI에게 입력되는 실제 프롬프트 예시]
"당신은 대한민국 건축심의위원회의 레드팀(반대 검토관)입니다. 아래 심의위원이 작성한 의견서 요약과 코멘트를 검토하고, 법적 리스크와 논리적 허점을 냉정하게 3가지 이상 지적하세요.

특히, 함께 제공되는 [통합검토 보고서 (RAG 사전 지식)]의 사업 기조(예: 기부채납을 통한 용적률 15% 완화 목표)를 훼손하는 설계적 오류가 있다면 가장 먼저 짚어주세요."

이와 같이 묻고 싶은 질문과 함께 괄호 쳐진 RAG 지식(사전 배경)을 함께 던져주는 것이 핵심 기술입니다.
